    Collaborative discussion Unit 1 - 3

    The majority of web applications consent the users to have immediate access to a series of services supplied by the organisation who provides the app. In most of the cases the user creates a profile to have access to the different information and services offered by the application like the opportunity of making bank transfers and check the own bank account in the case of the mobile banking apps, order and pay for the groceries shopping, and many others. These kind of web application can be subjected to identification and authentication failures which has been indicated as one of the top 10 web application security risks by the OWASP (OWASP, 2021).

    Considering the high values of information contained in these kind of web applications, cyber criminals might be interested in targeting them to gain access to valuable information by performing a series of authentication attacks. The attacks might result successful in case the applications have a series weaknesses like the exposure to session identifier in the URL using weak password and ineffective credential recovery processes, etc. (OWASP, 2021).

    Password can be stolen in different ways such as session hijacking where can be exploited a web session and in particular the attacker can manage to steal the session token sent from the web server to the user after completed the authentication (OWASP, 2020) . Another way is the use of credential stuffing subsequent to phishing or malware attack were the set of username and password stolen from a database can be used to access other websites (OWASP, 2020). These malicious attacks can be prevented by a series of measures like (OWASP, 2021):

  • Align password length, complexity, and rotation to NIST guidelines.
  • Ensure registration, credential recovery and API pathways are hardened against enumeration attacks.
  • Use secure session manager.
  • Implement weak password checks.

    • Collaborative discussion Unit 8 - 10

    In 2014 the author of TrueCrypt declared that the tool is not secured due to a series of unfixed security issues. In this occasion was advised to migrate from TrueCrypt to BitLocker (truecrypt.sourceforge.net, n.d.).

    The same year has been published the security assessment of TrueCrypt realised by iSec Partners for Open Crypto Audit Project. The report examined the bootloader and Windows kernel driver (Junestam & Guigo, 2014). Junestam and Guigo (2014) discovered eleven vulnerabilities when were analysing TrueCrypt with four of them with a medium severity, four with a low severity and three addressed to informational severity.

    The report stated “the source code for both the bootloader and the Windows kernel driver did not meet expected standards for secure code” (Junestam & Guigo, 2014). Moreover, were highlighted other issues such as the lack of comments, the use of insecure functions, weakness in the volume header integrity checks.

    Despite not presenting any critical issue among the vulnerabilities listed in the paper, I would not consider TrueCrypt safe to use because part of the code didn't meet the security standards, it present a series of medium vulnerabilities, bugs not being fixed and the software became obsolete considering it has been discontinued with the author abandoning the project and declaring the software insecure. As many of my peers highlighted in their blogpost, the obvious choice for TrueCrypt users was to move on a different encryption software such as VeraCrypt which is a fork of TrueCrypt.
